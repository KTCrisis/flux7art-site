Schema governance without vendor lock-in

Schema registries store schemas. They don't govern them. event7 adds a provider-agnostic governance layer above Confluent, Apicurio, Karapace, and Redpanda — with a Docker-first deployment that works out of the box.

If you're running event-driven systems in production, you have a schema registry somewhere. Confluent Schema Registry, Apicurio, Karapace, Redpanda — pick your flavor. It stores your Avro, JSON Schema, or Protobuf definitions. It checks compatibility when you push a new version. It does its job.

But at some point, someone asks a question the registry can't answer.

Who owns this schema? What business domain does it belong to? Which topics use it? Is it documented? Does it follow our naming conventions? When did it last change, and did that change break anything downstream?

The registry doesn't know. It was never designed to.

The governance gap

Schema registries are storage engines. They hold schemas, enforce compatibility modes, and serve versions over HTTP. That's it. Everything else — ownership, classification, business context, validation rules, documentation, cross-reference tracking — lives nowhere. Or worse, it lives in a spreadsheet, a Confluence page last updated eight months ago, or in someone's head.

Confluent recognized this and built Stream Governance. It's good. Tags, business metadata, data contracts, schema validation rules, audit logs. If you're all-in on Confluent Cloud, it solves the problem.

But that's the catch: if you're all-in on Confluent.

Plenty of teams run Apicurio behind their self-managed Kafka clusters. Others use Karapace on Aiven or Redpanda's built-in registry. Some run multiple registries across environments. Stream Governance doesn't help them. The governance metadata is locked inside Confluent's proprietary layer.

What event7 does

event7 sits above your registry as a governance layer. It connects to Confluent, Apicurio, Karapace, or Redpanda — reads your schemas, versions, and compatibility modes — and adds everything the registry doesn't provide.

Schemas stay in your registry. event7 never copies or duplicates them. It stores governance data in its own database: enrichments, channels, rules, AsyncAPI specs. If you disconnect event7, your registry is untouched.

Concretely:

  • Schema Explorer — browse subjects, versions, and formats across registries. Visual diff with field-level breaking change detection.
  • References Graph — interactive dependency graph. See which schemas reference which, spot orphans and hotspots, trace transitive chains (Order → Customer → Address).
  • Governance Rules — define naming conventions, required fields, documentation rules, compliance checks. Four types: conditions, transforms, validations, policies. Built-in templates, or import your Confluent Data Contract rules.
  • Schema Validator — validate before publishing. One report, three checks: registry compatibility, governance rules, visual diff. PASS, WARN, or FAIL.
  • Channel Model — map schemas to channels across 22 broker types. Kafka topics, RabbitMQ exchanges, Redis streams, Pulsar, NATS, Google Pub/Sub, AWS SNS/SQS. N:N bindings with strategy, role, and data layer tagging.
  • AsyncAPI — bidirectional. Import an existing spec into event7 (creates channels, bindings, enrichments in one click). Or generate AsyncAPI 3.0 specs from your event7 data. Drift detection included — event7 tracks when a schema changed but the spec wasn't regenerated.
  • Governance Scoring — three-axis score (enrichments + rules + schema quality) with a confidence indicator. The catalog shows a governance funnel: how many schemas are fully governed vs. partially vs. not at all.
  • EventCatalog Generator — export governance data to EventCatalog. Scores, rules, channels, teams, AsyncAPI specs. The first governance-aware generator for EventCatalog.

The Docker-first approach

event7 ships as a Docker Compose stack. One command, five containers, everything works:

git clone https://github.com/KTCrisis/event7.git
cd event7
docker compose -f docker-compose.local.yml up -d --build

You get:

That Apicurio instance matters. If you don't have a schema registry yet — maybe you're running RabbitMQ, or self-managed Kafka without a registry, or you just want to evaluate the tool — you have one out of the box. Connect it in Settings, start registering schemas, and governance is already active.

If you already have Confluent Cloud or an existing Apicurio, skip the bundled registry. Point event7 at your production registry instead. Same governance features, same UI, zero migration.

Two starting points

Starting from zero. You have events flowing through your broker but no schema registry. The Docker stack gives you Apicurio as a free, open-source registry (Apache 2.0). Register your schemas there, either manually through event7's UI or by importing an AsyncAPI spec. event7 handles governance from day one — no separate setup, no additional infrastructure.

Starting from Confluent. You already have hundreds of schemas in Confluent Schema Registry. Connect event7, and within seconds you're browsing your subjects with visual diffs, dependency graphs, and governance scoring. Your schemas don't move. event7 reads them through the Confluent API and stores governance data separately. You can even import your existing Confluent Data Contract rules (ruleSet + metadata) into event7's rule engine.

Both paths converge to the same place: a governed registry with enrichments, rules, channels, and AsyncAPI specs. The difference is whether the registry was already there or event7 brought one along.

Where it sits

event7 is not a registry. It's not a documentation portal. It's not a Kafka ops tool. It occupies a specific layer in the event-driven stack:

1  Infra / Broker         Kafka, RabbitMQ, Pulsar, NATS
2  Schema Registry        Confluent, Apicurio, Karapace, Redpanda
3  Governance             event7
4  Spec / Contract        AsyncAPI, CloudEvents
5  Documentation          EventCatalog, Backstage

Layer 3 is the gap. Most teams jump from layer 2 (registry) to layer 5 (documentation) — if they document at all. Governance and contract specs are either manual, vendor-locked, or missing entirely.

event7 fills layer 3 and bridges to layers 4 and 5. It generates AsyncAPI specs (layer 4) and exports to EventCatalog (layer 5). The governance data flows downstream automatically.

Provider-agnostic by design

Adding a new registry provider to event7 means writing one Python file that implements the SchemaRegistryProvider interface. No changes to services, routes, or frontend. The Apicurio provider was added exactly this way — one file, one factory branch.

Today, event7 supports Confluent Schema Registry (Cloud and Platform), Apicurio Registry v3, and any Confluent-compatible registry (Karapace, Redpanda, Red Hat Service Registry). AWS Glue and Azure Schema Registry are on the roadmap.

The governance data itself is completely decoupled from the registry. Enrichments, channels, rules, and AsyncAPI specs live in event7's PostgreSQL database. If you switch from Confluent to Apicurio tomorrow, your governance layer survives intact. That's the point.

What it looks like in practice

A typical daily workflow:

  1. Open the Schema Explorer. Browse your subjects, check recent version changes.
  2. Click a subject — see the schema, its evolution timeline, which channels use it, its governance score.
  3. Before pushing a new version, paste it in the Validator. Get a report: registry compatibility check + governance rules + visual diff. One screen, three signals.
  4. Check the Catalog for coverage gaps. Filter by "no enrichments" or "low governance score" to find schemas that need attention.
  5. Open the References Graph to understand impact before a breaking change. If Address is referenced by Customer which is referenced by Order, you see the full chain.
  6. Generate or refresh AsyncAPI specs. event7 detects drift — if a schema changed but the spec is stale, you see it immediately.

None of this requires leaving event7. None of it requires Confluent Cloud. And all of it works the same whether your schemas are in Confluent, Apicurio, or the bundled registry.

Open source
